Authentication Flow

Five auth methods with layered security
🔑

Email + Password

  • Payload built-in auth
  • 10 attempts / 15-min lockout
  • Email verification required
  • Cookie-based sessions (14 days)
📲

Two-Factor (TOTP)

  • AES-256-GCM encrypted secrets
  • HKDF key from PAYLOAD_SECRET
  • SHA-256 hashed backup codes
  • Backward-compatible w/ legacy
🤚

Passkeys (WebAuthn)

  • FIDO2 / biometric auth
  • Register + authenticate flows
  • Passkeys collection for storage
  • Phishing-resistant by design
✨

Magic Links

  • JWT-based tokens
  • Send → click → verify
  • Passwordless login option
🛡

Role System (Dual)

  • user.roles[] — admin, editor, viewer, crew_coordinator, shop_admin
  • user.crewRole — coordinator, member
  • Always use checkRole()
🚧

Middleware Guard

  • Checks payload-token cookie
  • Protects: /account, /chat, /crew, /schedule, /inventory, /shop
  • Redirects to /login

Architecture Decisions

Why things work the way they do
Why Payload CMS over a custom backend?
Payload provides auto-generated admin UI, built-in auth, file uploads, hooks, and access control out of the box. With 52 collections, hand-rolling CRUD would be unmanageable. Payload's hook system enables complex business logic (bidirectional sync, notifications) without a separate service layer.
Productivity
Why MongoDB over PostgreSQL?
The data model is document-oriented — crews, schedules, and chat messages are naturally nested. MongoDB's flexible schema supports rapid iteration on 52+ collections without migration overhead. Atlas M0 (free tier) keeps costs at zero during scaling.
Schema Flexibility
Why a dual role system?
user.roles[] = global permissions (admin, editor). user.crewRole = crew-scoped role. This separation allows a user to be a coordinator of one crew but a regular member globally. checkRole() unifies the check. CrewMemberships is the source of truth; crewRole is a computed convenience field.
Multi-Tenancy
Why polling instead of WebSockets for chat?
Vercel's serverless architecture doesn't support persistent WebSocket connections. Long polling via /api/chat/poll provides near-real-time updates within serverless constraints. The service worker's StaleWhileRevalidate strategy ensures snappy perceived performance.
Infrastructure Fit
Why IndexedDB for offline mutations?
Users often operate in low-connectivity environments (kitchens, warehouses). The offline queue stores up to 100 mutations in IndexedDB, replays them with exponential backoff when connectivity returns. This makes schedule sign-ups and inventory updates resilient to network drops.
Reliability
Why Stripe Connect for crew payouts?
Each crew operates semi-independently. Stripe Connect enables automatic fund splitting — when an order contains items from multiple crews, transfers are routed directly to each crew's connected account. Idempotency guards prevent double-transfers; refunds automatically reverse transfers.
Financial Isolation
Why AES-256-GCM for TOTP secrets?
TOTP secrets must be decryptable (unlike passwords). AES-256-GCM provides authenticated encryption — any tampering is detected. Key is derived from PAYLOAD_SECRET via HKDF-SHA256, meaning a single env var secures both Payload sessions and TOTP. Backward-compatible with legacy plaintext for migration.
Security
Why Cloudflare R2 over AWS S3?
Zero egress fees. R2 is S3-compatible (uses Payload's S3 adapter directly) but doesn't charge for data transfer. For a media-heavy app with avatars, chat attachments, inventory photos, and guide images across many crews, egress costs would add up fast on S3.
Cost Optimization
Why AbortController over cancelled flags?
AbortController is the browser-native way to cancel fetch requests. Unlike let cancelled = false flags (which only prevent state updates), AbortController actually terminates the HTTP request, freeing network resources. Stored in useRef to survive re-renders.
Performance
Why fire-and-forget in afterChange hooks?
Side effects (emails, notifications, cache revalidation) should never block the response. An inner async function with .catch() runs the work in the background while immediately returning the document. If a notification fails, the user still gets their response. Failed side effects are logged, not thrown.
Resilience

Patterns & Conventions

Rules that keep the codebase consistent
🔍

select + depth on every query

Always specify select: { field: true } and minimize depth (0=IDs, 1=direct). Never empty select: {} — it fetches everything.

🏠

Crew isolation in all queries

Every data query includes { crew: { equals: crewId } } to enforce multi-tenant data boundaries.

⚠

Every catch: log + toast

console.error + toast.error in every catch block. Never silently swallow errors. Server-side: payload.logger.error().

🔄

Context guards for hook loops

req.context?.skipFoo breaks bidirectional hook cycles (e.g., Users ↔ CrewMemberships sync).

🚫

Unconditional user fields

In hooks: data.user = req.user.id (never conditional). Prevents impersonation by overwriting any user-submitted value.

💰

Amounts in cents

order.amount stored as integers (cents). Display with (amount / 100).toFixed(2). Avoids floating-point errors.

🏷

extractId() for relationships

Relationship fields can be IDs or populated objects. extractId() normalizes to string ID regardless of depth.

📱

iOS-safe input sizing

text-base md:text-sm (16px on mobile) prevents iOS auto-zoom on input focus. maximumScale: 1 in viewport.

⏱

Rate limit all routes

rateLimit(key, limit, windowMs) on every API endpoint — both GETs and mutations. Prefer x-real-ip over forwarded.

🚀

Parallelize with Promise.all

Independent queries run in parallel. Hook side effects are fire-and-forget. useMemo for computed values in components.

  • Built with Serwist
  • Precaches app shell
  • Route-specific strategies
  • Custom cache keys

Caching Strategies

  • Schedule API — StaleWhileRevalidate (7d)
  • Recipes API — StaleWhileRevalidate (7d)
  • Inventory API — NetworkFirst (7d)
  • Chat channels — StaleWhileRevalidate (1d)
📴

Offline Queue

  • IndexedDB ocfcrews-offline
  • 100-mutation max capacity
  • Exponential backoff retry
  • States: pending → syncing → done
