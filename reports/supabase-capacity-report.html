Supabase Pro includes Supavisor — a server-side connection pooler (like PgBouncer) that multiplexes thousands of client connections into a small pool of actual PostgreSQL connections. Each Vercel serverless instance connects to Supavisor (port 6543, transaction mode) instead of directly to PostgreSQL. This completely eliminates the connection exhaustion problem.

Payload CMS adapter: @payloadcms/db-postgres — uses Drizzle ORM under the hood. Auto-creates tables, indexes, and junction tables from collection definitions.
Scaling path: Add Supabase Compute Add-on ($5–50/mo) for more CPU/RAM when needed. Or upgrade to Supabase Team ($599/mo) for SOC2 + priority support.

Bottom Line

With Supabase Pro (PostgreSQL + Supavisor pooling), Ably handling chat, and aggressive page caching (60s TTL), the app comfortably supports 5–10 crews of 30–50 members (~500 concurrent users) at ~$50/mo. The previous bottleneck — MongoDB Atlas Flex's 500-connection limit and 500 IOPS cap — is eliminated entirely. The new limiting factor is Vercel serverless concurrency (~1,000) and Supabase shared compute CPU, both addressable with low-cost add-ons. At 100 users, total cost is ~$95/mo ($0.95/user) — 25% less than the MongoDB setup which would have required an Atlas M10 upgrade ($127/mo).
