# Authentication API

## Overview

The Authentication API provides endpoints for logging in, inspecting the current session, refreshing tokens, and logging out. These endpoints are auto-generated by Payload CMS for the `users` collection and follow Payload's standard auth conventions.

**Endpoints:**

| Method | Path | Description |
|---|---|---|
| POST | `/api/users/login` | Authenticate and obtain a session |
| GET | `/api/users/me` | Get the currently authenticated user |
| POST | `/api/users/refresh-token` | Refresh an expiring JWT token |
| POST | `/api/users/logout` | Invalidate the current session |

**Source:** These endpoints are auto-generated by Payload CMS and are not defined in explicit route handler files. They are registered automatically when the `users` collection has `auth` enabled in its collection config (`src/collections/Users/index.ts`).

---

## Quick Start for Integrations

For programmatic or third-party integrations, follow this three-step flow:

### Step 1: Log In and Get a Token

```bash
curl -X POST https://your-domain.com/api/users/login \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "password": "your-password"}'
```

Response:

```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": { "id": "abc123", "email": "user@example.com", "name": "Jane Doe", ... },
  "exp": 1700000000
}
```

### Step 2: Use the Token in Subsequent Requests

Include the token in the `Authorization` header for all authenticated requests:

```bash
curl https://your-domain.com/api/users/me \
  -H "Authorization: JWT eyJhbGciOiJIUzI1NiIs..."
```

### Step 3: Refresh Before Expiry

Before the token expires (check the `exp` field), refresh it:

```bash
curl -X POST https://your-domain.com/api/users/refresh-token \
  -H "Authorization: JWT eyJhbGciOiJIUzI1NiIs..."
```

### JavaScript Fetch Example

```javascript
// Step 1: Login
const loginRes = await fetch('https://your-domain.com/api/users/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email: 'user@example.com', password: 'your-password' }),
});
const { token, user, exp } = await loginRes.json();

// Step 2: Use the token
const meRes = await fetch('https://your-domain.com/api/users/me', {
  headers: { Authorization: `JWT ${token}` },
});
const currentUser = await meRes.json();

// Step 3: Refresh before expiry
const refreshRes = await fetch('https://your-domain.com/api/users/refresh-token', {
  method: 'POST',
  headers: { Authorization: `JWT ${token}` },
});
const { refreshedToken } = await refreshRes.json();
```

## Endpoints

### Login

**Endpoint:** `POST /api/users/login`

#### Request

##### Headers

| Header | Required | Description |
|---|---|---|
| `Content-Type` | Yes | Must be `application/json` |

##### Request Body

```json
{
  "email": "string",
  "password": "string"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `email` | `string` | Yes | The user's email address |
| `password` | `string` | Yes | The user's password |

#### Response

##### Success Response

**200 OK**

```json
{
  "token": "string (JWT)",
  "user": {
    "id": "string",
    "email": "string",
    "name": "string",
    "roles": ["string"],
    "crew": "string | object",
    "...": "other user fields"
  },
  "exp": 1700000000,
  "message": "Auth Passed"
}
```

The response also sets an `HttpOnly` session cookie (`payload-token`) that the browser includes automatically on subsequent requests.

##### Error Responses

| Status | Error | Cause |
|---|---|---|
| 401 | `Unauthorized` | Invalid email or password |
| 400 | `Missing email or password` | Required fields not provided |

### Me (Current User)

**Endpoint:** `GET /api/users/me`

#### Request

##### Headers

| Header | Required | Description |
|---|---|---|
| `Cookie` | Yes | Payload session cookie (`payload-token`) |
| `Authorization` | Alt | `JWT <token>` for programmatic access |

No request body or query parameters required.

#### Authentication & Authorization

Any authenticated user can access this endpoint. It returns the user document associated with the current session or token.

#### Response

##### Success Response

**200 OK** -- Authenticated:

```json
{
  "user": {
    "id": "string",
    "email": "string",
    "name": "string",
    "roles": ["string"],
    "crew": "string | object",
    "collection": "users",
    "...": "other user fields"
  },
  "token": "string (current JWT)",
  "exp": 1700000000
}
```

**200 OK** -- Not authenticated:

```json
{
  "user": null
}
```

### Refresh Token

**Endpoint:** `POST /api/users/refresh-token`

#### Request

##### Headers

| Header | Required | Description |
|---|---|---|
| `Cookie` | Yes | Payload session cookie |
| `Authorization` | Alt | `JWT <token>` for programmatic access |

No request body required.

#### Authentication & Authorization

Must have a valid (non-expired) session or token.

#### Response

##### Success Response

**200 OK**

```json
{
  "refreshedToken": "string (new JWT)",
  "user": { "...": "user fields" },
  "exp": 1700000000,
  "message": "Token Refreshed"
}
```

The response also updates the `payload-token` cookie with the new token.

##### Error Responses

| Status | Error | Cause |
|---|---|---|
| 401 | `Unauthorized` | No valid session or token to refresh |

---

### Logout

**Endpoint:** `POST /api/users/logout`

#### Request

##### Headers

| Header | Required | Description |
|---|---|---|
| `Cookie` | Yes | Payload session cookie |
| `Authorization` | Alt | `JWT <token>` for programmatic access |

No request body required.

#### Response

##### Success Response

**200 OK**

```json
{
  "message": "You have been logged out successfully."
}
```

The response clears the `payload-token` cookie.

## Implementation Details

- **Token format:** Payload uses JSON Web Tokens (JWT) signed with the server's `PAYLOAD_SECRET`. The token encodes the user's ID, email, and collection.
- **Cookie name:** The session cookie is named `payload-token` and is set as `HttpOnly`, `Secure` (in production), and `SameSite=Lax`.
- **Token expiry:** Token lifetime is configured in the Payload config. Check the `exp` field (Unix timestamp) to know when the token expires.
- **2FA:** If a user has two-factor authentication enabled, the login flow may require an additional verification step via `/api/auth/2fa/verify`. See the 2FA documentation for details.
- **Browser vs. programmatic access:** Browsers use the `payload-token` cookie automatically. Programmatic clients (scripts, mobile apps, external services) should use the `Authorization: JWT <token>` header instead.
